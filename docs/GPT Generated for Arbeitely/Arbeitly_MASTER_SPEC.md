# 1. Product Overview

## 1.1 What Arbeitly Is

Arbeitly is a **hybrid SaaS + managed service platform** that helps candidates apply for jobs in Germany.

It has two major product modes:

### Free Plan
A self-service toolkit for candidates to:
- upload or create a CV
- edit and export CVs
- track their own job applications
- organize their job search

### Paid Plan
A managed service layer on top of the free product where:
- the candidate still keeps all free tools
- Arbeitly employees help apply to jobs on the candidate’s behalf
- AI is used to generate and tailor CVs and cover letters
- the candidate can transparently track everything happening

---

## 1.2 Core Product Philosophy

Arbeitly is **not just**:
- a job tracker
- a CV builder
- a job board
- an outsourcing agency dashboard

It is a **job application operating system** where:
- candidates get structure and visibility
- employees execute application work
- AI improves application materials
- admin controls the business logic and intelligence layer

---

## 1.3 Core Principle

**Admin defines the system → System assists with intelligence → Employees execute → Candidates receive full visibility**

# 4. Plan System

## 4.1 Free Plan

The Free Plan is the freemium acquisition layer.

Purpose:
- let candidate get value immediately
- build habit and engagement
- introduce core workflows
- create natural upgrade pressure

Included:
- CV builder
- CV editor
- PDF export with limits
- Files area for exported CVs
- Application tracker
- Profile page
- Activity log
- upgrade prompts

Not included:
- employee-managed execution
- AI enhancement
- AI-generated cover letters
- FAQ approval section
- job discovery pool access
- managed quota-based service

---

## 4.2 Paid Plan

The Paid Plan is the managed execution layer on top of the free plan.

**Key principle:**  
**Paid Plan = Free Plan + Managed Execution Layer**

Candidate still gets:
- CV section
- files
- applications
- profile

Additionally gets:
- onboarding questionnaire
- assigned employee
- managed application workflow
- AI-assisted CV generation
- AI-assisted cover letter generation
- FAQ / interview prep section
- application quota
- visibility into employee/system-added applications
- visibility into files generated by employee/system

---

## 4.3 Plans Are Dynamic

Paid plans are created dynamically by Super Admin.

Plan fields include:
- plan name
- price
- display price
- billing description
- total label for checkout
- optional price suffix (example: success fee wording)
- application quota / application limit
- included / excluded features
- "Most Popular" flag
- active / inactive status

The Free Plan is system-defined but can still have certain configurable behaviors (such as free export limit) controlled from admin settings.

---

## 4.4 Quota Rules

Quota applies only to paid candidates.

Key rules:
- quota belongs to candidate, not employee
- self-added applications by candidate do **not** consume quota
- employee/system-managed applications consume quota **only when status = Applied**
- “To Apply” does not consume quota
- “In Progress” does not consume quota
- “Failed” does not consume quota
- if quota is exhausted, employee cannot continue marking managed applications as “Applied”
- admin may increase quota manually
- candidate may buy add-on quota in future

Example:
- paid plan = 200 applications
- candidate currently = 145 / 200 used
- 55 applications remaining

## 7.13 Employee – Job Discovery (Central Pool)

This is one of the major updated architectural decisions.

### Purpose
A central pool of jobs accessible to all employees.

### Why It Changed
Earlier concept:
- per-candidate job discovery
- scraper or matching run candidate by candidate

Issue:
- too expensive
- duplicated work
- hard to scale

New approach:
- one central pool
- jobs dumped there manually and later by scraper
- employees look at pool and decide which jobs fit their assigned candidates

### Sources of Jobs in Pool
- manually added by employees
- manually added by admin
- future scraper ingestion
- future imports

### Employee View
Employee sees central pool and can view relevance against their assigned candidates.

For each job, the system can show:
- job title
- company
- location
- tags
- source platform
- language requirement
- match score per candidate

Example:
- Candidate A = 82%
- Candidate B = 67%

### Filtering
Employee can filter by:
- candidate
- match threshold
- role
- location
- source
- status / unused vs already actioned

### Bulk Action
Employee can:
- select multiple jobs
- select candidate(s)
- add them into that candidate’s Applications → To Apply

### Important Trigger Rule
**AI CV generation and cover letter generation should not happen just because a job exists in the pool.**

It should only happen when the employee decides the job should become an actual application candidate.

Trigger event:
**When job is added to candidate’s To Apply queue**

At that moment system should:
- determine best language route
- choose best base CV
- generate job-specific CV if needed
- generate cover letter
- attach outputs to application record
- save generated files into candidate Files

### Duplicate Handling
Pool should support duplicate detection / flags.
Actioning one job can optionally affect duplicate group handling.

### Matching Score Logic
Controlled by Super Admin prompt / matching logic.

## 8.9 Admin – Language Routing Rules

This is a critical Germany-specific feature.

Admin defines global rules for how CV language is chosen based on job language.

### Core Rules
- if job is English only → use EN CV
- if job is German only → use DE CV
- if job accepts English + German → generate/use DE version and prefer DE

### Auto-Generation Rule
If DE version does not exist:
1. system chooses best EN base CV
2. AI translates / adapts to DE according to admin logic
3. saves DE-tagged version
4. links it to the application

These are global rules, not per-candidate settings.

# 9. Job Discovery Architecture – Latest Direction

This is important enough to restate clearly.

## 9.1 Old Model
Per-candidate discovery queue.

Problem:
- expensive
- repetitive
- hard to scale

## 9.2 New Model
One central job pool.

Jobs enter pool from:
- employee manual addition
- admin manual addition
- future scraper ingestion

Employees then:
- browse central pool
- use filters and candidate match scores
- decide which jobs belong to which candidate
- move selected jobs into candidate To Apply queues

## 9.3 AI Generation Trigger
Generation does **not** happen at pool ingest time.

Generation happens only when:
**job is moved into candidate To Apply pipeline**

This reduces unnecessary AI cost and clutter.

# 15. Manual-First MVP Strategy

Current recommended build strategy:
- do **not** start by integrating scraper
- do manual job entry first
- central pool exists even without scraper
- employees/admin can manually add jobs
- moving to To Apply triggers generation
- validate workflow first
- plug scraper later

Why this is correct:
- less infrastructure risk
- less data inconsistency
- validate real product before external automation
- easier testing
- cheaper to build and iterate

# 18. Final Summary

Arbeitly is a **hybrid job application operating system for Germany-focused job seekers**.

It combines:
- a freemium self-service candidate toolset
- a paid managed application workflow
- an employee execution layer
- an admin-controlled AI intelligence layer
- a central job discovery pool
- language-aware CV routing
- transparent shared visibility between candidate and service provider

In simple terms:

- **Free candidate** gets a useful job toolkit
- **Paid candidate** gets that toolkit plus managed execution
- **Employee** works from assigned candidates and a central job pool
- **Admin** controls plans, prompts, assignment, rules, and performance
- **AI** assists with matching, enhancement, generation, and routing
- **Quota** monetizes managed execution cleanly and only when real application work is completed

This document should be treated as the primary product spec until replaced by a newer master version.
